# Claude Code 플러그인, 진짜 도움이 될까? oh-my-claudecode Hook A/B 테스트

**한줄 요약**: Claude Sonnet 4.6으로 동일한 코딩 태스크를 6번 실행했다 (OMC hook ON 3회, OFF 3회). 동적 hook은 코드 품질과 비용에 유의미한 차이를 만들지 못했다. 하지만 토큰이 어디로 가는지 추적하면 흥미로운 이야기가 나온다.

> [English version (README.md)](./README.md)

## 이게 뭔데?

[oh-my-claudecode (OMC)](https://github.com/yeachan-heo/oh-my-claudecode)는 Claude Code용 플러그인으로, 동적 hook, 전문 에이전트, MCP 도구, 워크플로우 스킬을 추가한다. 세션 시작, 도구 사용 전후, 컴팩션 등 다양한 시점에 `system-reminder` 태그로 컨텍스트를 주입한다.

너무너무 궁금했다. **동적 hook이 실제로 Claude가 더 나은 코드를 쓰게 만드는가, 아니면 그냥 토큰을 태우고 있는가?**

## 실험 설계

### 태스크

**마크다운 에디터**를 처음부터 구현하는 멀티파일 코딩 태스크
- `index.html` — 에디터 UI (툴바, 미리보기 패널, 분할 뷰)
- `style.css` — 반응형 레이아웃, 구문 강조
- `markdown-parser.js` — 풀 파서 (제목, 리스트, 코드 블록, 테이블, 링크, 이미지 등)
- `editor.js` — 툴바 액션, 실시간 미리보기, 드래그 리사이즈, 키보드 단축키

### 3개 순차 프롬프트 (같은 세션, `--continue` 사용)

| 프롬프트 | 태스크 |
|---------|--------|
| **P1** | 4개 파일 전체 구현 |
| **P2** | 종합 테스트 작성 (`editor.test.js`) 후 실행 |
| **P3** | 버그 리뷰 (XSS, 엣지케이스, 메모리 누수, 접근성), 수정, 재테스트 |

### 실험 조건

| | Hook ON | Hook OFF |
|---|---|---|
| **CLAUDE.md 정적 콘텐츠** | 로드됨 | 로드됨 |
| **MCP 도구 (32개)** | 등록됨 | 등록됨 |
| **스킬 & 에이전트 카탈로그** | 사용 가능 | 사용 가능 |
| **동적 hook 실행** | **활성** | **비활성** |

`DISABLE_OMC=1` 환경변수는 OMC bridge.js에서 모든 hook 실행을 건너뛴다 (`return { continue: true }`). 그 외 모든 것은 동일 — 같은 시스템 프롬프트 가중치, 같은 도구 정의.
- `DISABLE_OMC=1` 값은 https://yeachan-heo.github.io/oh-my-claudecode-website/docs.html#troubleshooting 이 페이지에서 확인했다.

**즉, 이 실험은 "OMC 동적 hook의 독립적 효과"만을 측정한다.**

### 실행 방법

```bash
claude -p --model sonnet --dangerously-skip-permissions --output-format json "$PROMPT"
```

- 조건당 3회, 총 6회 실행
- 6개 모두 같은 머신에서 동시 병렬 실행
- 프롬프트별 타임스탬프 기록
- 전체 transcript 보존

### 프롬프트별 캐시 읽기 토큰 (여기가 핵심)

| 프롬프트 | Hook OFF | Hook ON | 차이 |
|---------|----------|---------|------|
| **P1 (구현)** | 195K | 424K | **+117%** |
| **P2 (테스트)** | 639K | 1,021K | +60% |
| **P3 (리뷰)** | 2,135K | 1,264K | -41% |

**P1이 핵심이다.** Hook ON 상태에서 SessionStart hook이 컨텍스트 브리핑을 주입하고, 이것이 매 턴마다 캐시로 반복 읽힌다. 첫 번째 프롬프트에서 캐시 읽기 토큰이 2배. Hook이 이 태스크에 필요 없는 컨텍스트를 쏟아붓고 있다.

그런데 P3에서는 Hook OFF가 오히려 캐시를 *더 많이* 읽는다. 설명이 안 된다.

## 동적 Hook이 실제로 하는 일

OMC hook이 활성 상태일 때 세션 중 발생하는 이벤트

| Hook | 시점 | 주입 내용 |
|------|------|----------|
| **SessionStart** | 세션/컴팩트 | 컨텍스트 브리핑 — 최근 작업, worktree 정보, OS 상세 |
| **PreToolUse** | Edit/Write 전 | 전략적 컴팩션 제안 (도구 호출 카운터) |
| **PreCompact** | `/compact` 전 | 체크포인트 + 프로젝트 메모리 보존 |
| **PostToolUse** | 각종 도구 후 | "병렬 실행 사용하세요" 리마인더, 백그라운드 작업 알림 |

transcript 파일에서 `system-reminder` 태그로 직접 확인할 수 있다.

## 이 실험이 증명하지 않는 것

솔직하게 말하면 — 이건 좁은 범위의 실험이다. 이 실험으로 **말할 수 없는** 것

- **"OMC가 무의미하다"** — hook만 테스트했다. MCP 도구 (LSP, AST grep, 노트패드, 상태 관리), 에이전트 카탈로그, 스킬은 양쪽 모두 로드되었고, 그 효과는 미측정이다.
- **"Hook이 절대 도움 안 된다"** — 단일 세션, 단일 태스크 테스트였다. 멀티세션 워크플로우에서 컨텍스트 지속성이 중요한 경우 hook이 빛날 수 있다.
- **"Opus에서도 동일하다"** — Sonnet만 테스트했다. Opus는 주입된 컨텍스트를 다르게 활용할 수 있다.
- **"모든 태스크에 적용된다"** — 4파일 에디터는 중간 복잡도다. 20개 이상 파일에 걸친 대규모 리팩토링이나 팀 협업은 다를 수 있다.

## 핵심 발견

1. **OMC 동적 hook은 단일 코딩 태스크의 코드 품질을 개선하지 않았다.** hook ON/OFF 양쪽 모두 코드 품질 15.0/20, 비용 $5.56, 생성 코드 라인 ~2,430으로 동일했다. Claude Sonnet은 정적 CLAUDE.md 지시문만으로 이미 충분히 동작한다.

2. **OMC SessionStart hook은 첫 프롬프트(P1)에서 캐시 읽기 토큰을 2배로 늘린다.** hook ON 시 P1 캐시 읽기가 424K, hook OFF 시 195K였다. SessionStart hook이 주입하는 컨텍스트 브리핑이 매 턴 캐시로 반복 소비되기 때문이다. 캐시 읽기 단가가 $0.30/1M으로 저렴하지만, 대규모 운용 시 무시할 수 없는 비용이 된다.

3. **hook이 만드는 토큰 오버헤드는 세션이 길어질수록 희석된다.** P3까지 누적하면 hook ON/OFF 간 총 비용 차이가 $0.00이다. 세션이 길어지면 hook 주입분의 비중이 줄어들기 때문이다.

4. **동일 조건 내 run 간 비용 분산이 hook ON/OFF 간 차이보다 훨씬 크다.** hook을 끈 OFF 조건에서도 run1은 $3.51, run3은 $6.93으로 같은 조건 안에서 비용이 2배 벌어진다. hook이 만드는 차이가 설령 존재하더라도, 이 정도 분산 앞에서는 통계적으로 유의미하지 않다.

5. **OMC의 실제 오버헤드는 동적 hook이 아니라 정적 시스템 프롬프트에 있다.** OMC가 등록하는 32개 MCP 도구 정의 + 에이전트 카탈로그 + 스킬 목록은 `DISABLE_OMC=1` 여부와 관계없이 매 턴 시스템 프롬프트에 포함된다. 이 상수 비용이 진짜 세금이고, 이번 실험은 그 위에서 hook의 추가 효과만 측정한 것이다.

## 레포 구조

```
omc-hook-experiment/
├── README.md                # 보고서 (영어)
├── README_kr.md             # 보고서 (한국어)
├── data/
│   ├── enable-omc/          # OMC hook 활성 상태
│   │   ├── run1/            # metrics.json, out*.json, 생성된 소스코드
│   │   ├── run2/
│   │   └── run3/
│   └── disable-omc/         # OMC hook 비활성 (DISABLE_OMC=1)
│       ├── run1/
│       ├── run2/
│       └── run3/
├── transcripts/              # 전체 Claude Code 세션 기록 (JSONL)
│   ├── enable-omc-run1/     # 메인 세션 + 서브에이전트 transcript
│   ├── enable-omc-run2/
│   ├── enable-omc-run3/
│   ├── disable-omc-run1/
│   ├── disable-omc-run2/
│   └── disable-omc-run3/
├── scripts/
│   ├── run-all.sh           # 6개 run 병렬 실행
│   ├── run-single.sh        # 단일 run (프롬프트별 타임스탬프 기록)
│   ├── capture.sh           # transcript 수집 + 메트릭 추출
│   ├── extract_metrics.py   # 토큰/비용/품질 메트릭 추출
│   └── analyze.py           # 조건 간 비교 분석
└── prompts/
    ├── prompt1.txt           # "마크다운 에디터를 만들어라..."
    ├── prompt2.txt           # "테스트를 작성하고 실행해라..."
    └── prompt3.txt           # "리뷰하고, 버그 고치고, 재테스트..."
```

### Run별 데이터 파일

- `metrics.json` — 추출된 메트릭 (토큰, 비용, 시간, 도구 분석)
- `out1.json`, `out2.json`, `out3.json` — 프롬프트별 Claude CLI 원시 출력 (`usage` 블록 포함)
- `*.js`, `*.html`, `*.css` — 생성된 소스코드 (실제 에디터)
- `.start_ts`, `.end_ts`, `.p*_ts` — 소요 시간 계산용 Unix 타임스탬프
- `err*.log` — Claude CLI stderr (보통 비어있음)

### Transcript 형식

`~/.claude/projects/`의 JSONL 파일. 각 줄은 대화 턴을 나타내는 JSON 객체
- `role` (user/assistant/system)
- `content` (메시지 텍스트, 도구 호출, 도구 결과)
- `usage` (턴별 토큰 수)
- `system-reminder` 태그 (hook 주입 내용이 여기에 보임)

## 실행 환경

- **모델**: Claude Sonnet 4.6 (`claude-sonnet-4-6`)
- **OMC 버전**: 4.7.6

- **머신**: macOS (Darwin 24.6.0), Apple Silicon
- **6개 run 모두 같은 머신에서 동시 실행**
